# A 股挖因子系统路线图 v1

日期：2026-06-24

## 1. 定位

这份路线图定义：如何把当前 `Factor Autoresearch` 的研究毛坯，推进成第一版“可以上线挖因子”的 A 股因子挖掘系统。

这里的“上线挖因子”不是指实盘交易上线。它指的是：

- 可以持续生成候选因子。
- 可以稳定计算和评价候选因子。
- 可以做样本外和滚动验证。
- 可以识别重复因子和相似因子。
- 可以把有证据的因子沉淀到资产库。
- 可以让 agent 按固定规则持续迭代。

它暂时不包括：

- 实盘交易。
- 下单执行。
- 组合构建。
- 生产级风控。
- 券商接口。
- 资金分配。

当前仓库已经有一个能跑的研究 sandbox：

- 固定数据集：每次研究读取同一份冻结数据，避免临时改口径。
- 候选因子 JSONL：用一行一个 JSON 的方式记录候选因子。
- DSL 校验：检查表达式是否只用了允许字段和允许函数。
- 因子计算：把候选表达式算成因子值。
- 预处理：去极值、标准化、行业和市值中性化。
- IC / RankIC / 分层收益：衡量因子和未来收益的关系。
- candidate gate：候选因子的第一层通过规则。
- run artifacts：每次运行的结果产物。
- candidate registry：通过 gate 的候选记录。
- research notes / memory：研究笔记和长期经验。

下一阶段的目标，是把它从“能跑一次”升级成“能持续挖、能筛、能沉淀”的系统。

## 2. 总体区块

后续推进不建议拆得太细。第一版路线图按 5 个大区块走：

```text
1. 计算与评价引擎
2. 数据与样本协议
3. 因子验收与稳健性 gate
4. 因子资产库
5. Agent 挖掘闭环
```

核心顺序：

```text
先跑得动，再判得准，再沉淀得住，最后自动挖。
```

## 3. 区块一：计算与评价引擎

目标：让候选因子评价足够快、足够稳定，能支撑更大的候选批次和多轮验证。

这一块包括：

- `PanelStore`：矩阵化面板数据结构，把 `(日期 x 股票)` 的数据放到更适合高速计算的结构里。
- 表达式 DAG / cache：把表达式拆成可复用的计算图，重复子表达式只算一次。
- NumPy / Numba kernels：用更底层的数组计算替代热点路径里的 pandas 慢操作。
- 预处理加速：加速 winsorize、zscore、中性化。
- metrics / diagnostics 加速：加速 IC、RankIC、分层、诊断统计。
- 候选级并行：多个候选因子可以同时算。
- legacy / v1 等价验证：新旧引擎必须算出一致结果，不能为了快牺牲语义。
- legacy fallback：新引擎出问题时可以回退旧路径。

为什么它排第一：

- OOS（样本外）会让一次评价拆成多段。
- walk-forward（滚动前推验证）会重复跑多个时间窗口。
- 相关性去重需要读取和比较很多因子值。
- agent 后面会生成更大批次候选。
- 如果计算不先提速，后面的验证层会很难迭代。

验收标准：

- 当前候选批次达到目标速度。
- legacy 和 v1 的 gate 结果一致。
- 输出产物格式不破坏现有读取方式。
- 新引擎可以安全切为默认评价路径。

已有主文档：

- `docs/framework/factor-autoresearch-compute-engine-v1-spec.md`

## 4. 区块二：数据与样本协议

目标：保证研究样本干净、可复现，并尽量避免未来函数。

这一块包括：

- 股票池协议：明确研究 universe，即每个日期哪些股票可以参与评价。
- PIT 数据要求：point-in-time，按当时可见数据来研究，避免使用未来才知道的信息。
- 基础过滤：ST、停牌、涨跌停、低流动性、低成交额等。
- forward return 定义：明确因子信号日、进场价、退出价和持有期。
- 样本切分：明确训练段、验证段、样本外段。
- OOS 窗口：out-of-sample，用没参与筛选的后续样本检验因子。
- walk-forward 窗口：滚动前推验证，用过去一段筛选，再看未来一段是否还有效。
- dataset manifest：记录数据来源、日期、字段、过滤规则和版本。
- 更大的 A 股研究股票池：从当前中证500 sandbox，逐步扩到更适合挖因子的 A 股范围。

关键判断：

OOS 和 walk-forward 不只是 gate 里的几个指标。它们首先是“样本协议”。日期怎么切、窗口怎么滚、forward return 怎么定义，都必须固定下来，后面的 gate 只是消费这些切片。

第一版建议：

- 保留 `sandbox_v1` 作为快速开发数据集。
- 增加 `mining_v1` 样本协议，专门用于严肃挖因子。
- 定义时间顺序切分，避免随机打乱。
- 定义 formation window：用于观察和筛选的历史窗口。
- 定义 validation / OOS window：用于验证筛出来的候选是否还有效。

验收标准：

- 每次 run 的 manifest 都记录样本协议。
- 候选因子不能私自改变日期范围、股票池或 forward return。
- OOS 和 walk-forward 切片可复现。
- metrics、diagnostics、gate 都能复用同一套样本切片。

已有主文档：

- `docs/framework/factor-autoresearch-data-sample-protocol-v1-spec.md`

## 5. 区块三：因子验收与稳健性 gate

目标：把现在的单次全样本 gate，升级成上线挖因子前的资格审查层。

这一块包括：

- 当前 gate：coverage、IC、RankIC、单调性、复杂度这些基础门槛。
- OOS 检查：看样本外是否还有效。
- walk-forward 检查：看多个滚动窗口里是否稳定。
- 年份诊断：避免只在某一年有效。
- 行业诊断：避免只靠某几个行业贡献。
- 市场状态诊断：有数据后再看牛熊、震荡、风格切换等状态。
- 因子相关性去重：避免把同一个信号换皮重复收进库。
- 残差增量检验：看新因子剔除已有因子影响后，是否还有新增信息。
- turnover proxy：换手代理指标，观察因子组合是否过度频繁变化。
- cost proxy：成本代理指标，粗略判断交易成本是否可能吃掉收益。
- capacity / liquidity proxy：容量和流动性代理指标，避免信号只存在于难交易股票。
- 晋升规则：从 candidate 到 watch，再到 accepted 或 official candidate。

关键判断：

OOS、walk-forward、相关性去重、稳健性诊断，不应该做成一堆零散工具。它们应该合在一个“因子验收区块”里，最终输出一份统一 acceptance report。

第一版建议：

- 保留当前 gate，命名为 `in_sample_gate`。
- 增加 `oos_gate`：样本外门槛。
- 增加 `walk_forward_gate`：滚动窗口稳定性门槛。
- 增加 `correlation_dedup_gate`：相关性去重门槛。
- 增加 `incremental_signal_gate`：新增信息门槛。
- 所有失败规则统一写进 acceptance report。

相关性去重分三层：

- 精确去重：表达式 hash 一样，直接认为重复。
- 暴露去重：新因子和 registry 中已有因子的平均绝对相关性过高，则认为高度相似。
- 增量去重：把新因子对已有因子做残差化，再看残差因子是否还有 IC / RankIC。

验收标准：

- 一个因子可以通过样本内 gate，但因为 OOS 或 walk-forward 不稳而不进入最终 registry。
- registry 只记录通过最终验收的因子。
- 每个 accepted 因子都带样本内、样本外、滚动窗口的证据。
- 重复或近似重复因子会被拒绝，或者标记为已有因子的 variant。

## 6. 区块四：因子资产库

目标：把通过验收的候选，管理成可追踪、可复核、可淘汰的研究资产。

这一块包括：

- candidate registry：候选注册表。
- watchlist：观察名单，记录还需要继续验证的因子。
- accepted factors：已通过验收的因子。
- official factor candidates：准备进入更正式流程的因子。
- 因子版本：同一因子不同表达式或不同参数的版本记录。
- expression hash：表达式哈希，用于识别重复和追踪版本。
- run lineage：因子来自哪些候选、哪些 run、哪些数据协议。
- factor family：因子家族，例如波动类、反转类、成交量类。
- correlation cluster：相关性簇，把高度相似的因子归到一组。
- 状态流转：candidate、watch、accepted、retired 等。
- review notes：人类或 agent 对因子的复核记录。
- retirement / invalidation：因子失效、重复、被替代后的退役记录。

建议状态：

```text
candidate              候选
watch                  观察中
accepted               已验收
official_candidate     官方候选
retired                已退役
rejected_duplicate     因重复被拒
rejected_unstable      因不稳定被拒
```

关键判断：

registry 不应该只是“一轮跑赢的因子列表”。它应该成为系统对因子资产的记忆：这个因子为什么被接收、哪里失败过、和谁相似、是否还有效。

验收标准：

- 每个 accepted 因子都能追溯到候选、run、样本协议和验收报告。
- 每个因子都有当前状态。
- 可以查看因子家族和相关性簇。
- 晋升依赖多轮证据，而不是单次好结果。

## 7. 区块五：Agent 挖掘闭环

目标：在评价和验收系统可靠之后，让 agent 按固定规则持续挖因子。

这一块包括：

- 读取 memory、research notes、registry 和最新 runs。
- 生成候选批次。
- 避开历史上反复失败的方向。
- 对已通过或接近通过的因子家族做变体。
- 自动运行 validate 和 evaluate。
- 读取 acceptance report。
- 写入 research notes。
- 只在多轮稳定后提出 memory 更新。
- 定期重跑已接收因子，观察是否失效。

关键判断：

Agent 闭环要放在验收层之后。否则 agent 会很快生成大量候选，但系统还没有足够成熟的判断层，容易把样本内噪音沉淀成“成果”。

验收标准：

- agent 能完整跑一批候选，但不能改数据、gate、config 规则。
- 每一批都留下可读研究笔记。
- 下一轮建议基于验收证据，而不是只看单次排名。
- 长期 memory 和单轮噪音分开。

## 8. 推荐推进顺序

正式推进顺序：

```text
1. 计算与评价引擎
2. 数据与样本协议
3. 因子验收与稳健性 gate
4. 因子资产库
5. Agent 挖掘闭环
```

前三个区块完成后，系统才算具备严肃挖因子的基础。

后两个区块完成后，系统才算开始具备可持续、可沉淀、可半自动运行的能力。

## 9. 里程碑

### 里程碑 A：快速可跑实验室

范围：

- compute engine v1
- legacy 等价验证
- metrics 和 diagnostics 加速

结果：

- 系统可以快速评价更大的候选批次。

### 里程碑 B：严肃评价实验室

范围：

- 数据与样本协议
- OOS 窗口
- walk-forward 窗口
- acceptance report 格式

结果：

- 系统可以区分“样本内看起来好”和“样本外仍然能用”。

### 里程碑 C：可上线挖因子实验室

范围：

- 最终验收 gate
- 因子相关性去重
- 残差增量检验
- watchlist / accepted 状态

结果：

- 系统可以持续挖掘并保留非重复、有证据的候选因子。

### 里程碑 D：Agent 运营的挖掘闭环

范围：

- 候选生成
- 批次循环
- 研究笔记自动化
- memory 更新协议
- 周期性复核

结果：

- agent 可以持续提出、测试、总结和改进因子想法。

## 10. 本路线图不覆盖

这份路线图不覆盖：

- 实盘交易。
- 下单执行。
- 组合构建。
- 生产风控。
- 券商接口。
- 日内执行。
- 资金分配。

这些应该等挖因子系统稳定后，另开交易系统路线图。

## 11. 工作原则

最终原则：

```text
先跑得动，再判得准，再沉淀得住，最后自动挖。
```

展开说：

- 跑得动：计算引擎能支撑批量候选和多窗口验证。
- 判得准：样本外、滚动窗口、去重和稳健性检查足够严格。
- 沉淀得住：因子资产库能记录证据、版本、状态和失效原因。
- 自动挖：agent 在固定规则内持续生成、测试、总结和迭代。
