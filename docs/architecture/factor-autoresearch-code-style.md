# Factor Autoresearch Code Style

这份文档沉淀 `factor_autoresearch/` 代码风格迁移中已经验证可用的写法。
目标不是追求“好看”，而是让后续改动更容易扫读、更少把边界逻辑和计算逻辑重新缠在一起。

## 1. 总体原则

- 先整理阅读结构，再考虑行为重构。
- 先把模块职责说清楚，再展开实现细节。
- 纯计算函数重点表达“怎么算”。
- IO、配置、删除、外部输入校验重点表达“哪里需要防御”。
- 流程型模块重点突出主路径，不把所有细节堆在一个长函数里。

## 2. 模块说明

每个核心模块开头都应有模块级 docstring，用 2 到 5 行说明：

- 这个模块负责什么。
- 它不负责什么。
- 如果存在稳定命名约定，也放在这里。

推荐写法：

```python
"""
算子库模块
定义因子 DSL 使用的基础算子与注册表。
命名约定：
- 二元基础算子使用 add/sub/mul/div
- 时间序列算子使用 ts_ 前缀
- 截面算子使用 cs_ 前缀
"""
```

如果模块本身就是边界层，也应直接写明，例如“负责读取 dataset 并做合同校验”或“负责 run 目录落盘，不承载业务计算”。

## 3. 分区标题

文件内部按职责分区，使用统一分隔线：

```python
# ============== 基础辅助函数 ==============
```

分区标题只解决一件事：帮助读者快速定位。

推荐做法：

- 先放数据结构，再放辅助函数，最后放主入口。
- 同一分区内的函数应服务同一类职责。
- 分区标题使用业务词，不使用“其他”“杂项”。

常见分区示例：

- `基础辅助函数`
- `指标结果`
- `候选指标计算`
- `字段合同`
- `加载器`
- `结果写入`

## 4. Docstring 写法

类和主要函数都写简短中文 docstring，优先用下面这种格式：

```python
"""功能名: 一句话说明。"""
```

例如：

```python
"""安全除法: 执行 x / y，并把除零结果处理为 NaN。"""
"""按股票分组: 供时间序列算子复用。"""
"""算子描述: 记录调用类型、参数个数和实现函数。"""
```

约束：

- 一般一行够用，不够再补第二行。
- 先说职责，再说关键约束。
- 不重复类型标注里已经写清楚的信息。
- 私有但阅读成本高的辅助函数，也值得补一句。

不推荐：

- 空泛描述，如“处理数据”“执行逻辑”。
- 把实现细节全部塞进 docstring。
- 英文、中文、缩写混杂到难扫读。

## 5. 命名规则

命名按场景选，不强求全仓统一长名或短名。

短命名适合：

- 算子实现
- 数学推导
- 局部窗口计算
- 生命周期很短的中间变量

例如：`x`、`y`、`d`、`g`、`v`

长命名适合：

- CLI 参数
- IO 路径
- 配置对象
- 跨函数传递的重要结果
- 流程型模块里的主路径变量

例如：`candidate`、`dataset`、`config`、`metrics_result`、`registry_path`

补充约束：

- 公共入口参数优先可读，不为了“统一简短”牺牲语义。
- 同一函数里不要混用多个意义接近的缩写。
- 已经是领域固定词的命名直接保留，如 `rankic`、`horizon_rows`、`in_universe`。

## 6. 防御性代码放哪里

防御性代码应放在边界层，而不是铺满所有内部函数。

应该保留防御的地方：

- CLI 参数入口
- JSON / TOML / Parquet 读取
- manifest / schema 校验
- 文件写入和删除
- 空数据、缺字段、重复主键
- 除零、零标准差、空截面等数学边界

应该尽量减少的地方：

- 已由上游保证合同的内部纯函数
- 同一件事在多层重复校验
- 为兼容接口硬塞的无意义兜底参数
- 让主逻辑变难读的大段保护分支

判断标准很简单：

- 外部输入不可信，就在入口挡住。
- 内部合同已经稳定，就让主逻辑保持直线。

## 7. 纯计算函数写法

纯计算函数优先保持“短、直、可验证”。

推荐做法：

- 输入输出尽量稳定，不顺手夹带写文件、打日志、改全局状态。
- 主体代码直接表达公式或步骤。
- 局部辅助函数只服务当前计算语义。
- 遇到数学边界，只保留必要保护，不重复做上游已经做过的校验。

例如，下面这种写法是推荐方向：

```python
def ts_return(x: pd.Series, d: int) -> pd.Series:
    """收益率: x 相对 d 天前取值的变化比例。"""
    g = _by(x)
    return div0(x, g.shift(d)) - 1.0
```

如果函数偏长，可以做两件事：

- 提炼少量局部段落变量，让阶段更清楚。
- 把独立步骤拆成私有辅助函数，但不要拆成过度跳转。

## 8. 流程型模块写法

`evaluate.py`、`prepare.py`、`cli.py` 这类模块不应只靠“补 docstring”解决可读性问题。

这类模块的重点是：

- 主路径一眼能扫出来。
- 步骤函数职责清楚。
- 边界动作和业务计算分开。
- 异常处理不打断主线阅读。

推荐结构：

1. 模块说明先交代流程入口职责。
2. 对外入口函数保持薄。
3. 把加载、校验、计算、落盘、汇总拆成步骤函数。
4. 复杂分支放进命名清楚的辅助函数，不在主流程里堆细节。

不推荐：

- 一个超长函数同时处理读取、计算、落盘、日志和异常收口。
- 把大量边界判断直接插在主路径中间。
- 为了“少函数”把可复用步骤重新写散。

## 9. 迁移时的取舍

风格迁移默认只做三类事：

- 补模块说明。
- 补分区标题和 docstring。
- 清理明显影响阅读的命名和重复防御。

默认不做的事：

- 改公开函数签名。
- 改指标口径、落盘格式、字段合同。
- 借风格迁移顺手重写业务逻辑。

如果需要改行为，应单独立项，不混在代码风格迁移里。
