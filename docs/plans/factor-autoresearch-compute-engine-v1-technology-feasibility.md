# Factor Autoresearch Compute Engine v1 技术可行性说明

## 1. 这份文档解决什么问题

这份文档对应 Compute Engine v1 的 Task 9。

目标不是否定 `Polars`、`DuckDB`、`GPU`，而是回答一个更直接的问题：

- 现在要不要把它们放进 v1 runtime（运行时主链路）？
- 如果现在不放，适合放在哪？
- 后面什么条件满足后，再重新评估？

结论先说在前面：**v1 暂不把 Polars、DuckDB 计算路径、GPU 计算引入 runtime 主链路。**

原因很简单：当前 v1 最重要的是先把 `legacy / v1` engine 路由、可复现执行、稳定产出顺序、错误可审计这条链路做稳，而不是同时引入第二套甚至第三套计算语义。

## 2. 📌 结论

- ✅ `Polars`：现在**不进入 runtime 主链路**
- ✅ `DuckDB`：现在**不进入 runtime 主链路的指标计算/因子计算路径**
- ✅ `GPU`：现在**不进入 runtime 主链路**

当前建议：

- `runtime canonical path`（唯一口径主路径）继续保持简单、可测试、可解释
- 并发先用 `ThreadPoolExecutor` 解决“多任务调度和顺序稳定”问题
- 新技术先放在“离线分析、预处理、实验性加速、诊断工具”这些边界位置

## 3. 🧩 为什么现在不进 runtime

### 3.1 Polars 暂不进入 runtime 的原因

`Polars` 的优点很明确：

- 列式执行快
- 内存效率通常更好
- lazy execution（延迟执行）对复杂数据管线很有吸引力

但 v1 当前不适合直接引入，主要有 4 个原因：

1. **会增加第二套数据语义。**
   当前 engine v1 还在搭基础边界。此时如果主链路同时支持 pandas/Polars 两套行为，后续一旦出现空值、排序、分组窗口、类型提升差异，就会让结果解释和排障成本明显上升。

2. **测试成本会翻倍。**
   现在最需要验证的是“相同输入 -> 相同顺序 -> 相同结果”。如果主路径引入 Polars，就需要补一整套行为一致性测试，而这不是本阶段最值钱的投入。

3. **很多收益现在拿不到。**
   v1 当前新增的是 engine 路由和并发调度，不是超大规模列计算。瓶颈是否真的在 DataFrame 引擎上，现阶段没有证据。

4. **会弱化故障定位。**
   现在我们更需要“出了错就知道是哪一个 candidate、哪一层、什么顺序出的问题”。Polars 不是不能做，而是会把诊断面再拉宽一层。

### 3.2 DuckDB 暂不进入 runtime 计算链路的原因

`DuckDB` 非常适合查 parquet，也适合做本地分析。

但 v1 暂时不把它放进 runtime 主链路里的因子计算/指标计算部分，原因是：

1. **它更适合作为查询引擎，不适合作为当前唯一计算语义。**
   如果 runtime 一部分在 Python/线程里算，一部分转成 SQL/duckdb 算，行为边界会变复杂。

2. **表达式迁移成本高。**
   当前 worker 模型是“输入 item -> 调 Python worker -> 拿回结果”。如果把运行时核心逻辑搬到 DuckDB，很多计算需要重新表达成 SQL 或混合层，不利于先把 v1 路由层做稳。

3. **错误模型会更分裂。**
   现在异常可以和输入顺序一一对应。加入 DuckDB runtime 后，错误可能变成 SQL 编译错误、执行错误、类型错误、桥接错误，排障路径更长。

4. **没有解决这次任务的核心问题。**
   这次任务重点是：
   - engine 名称校验
   - `jobs=auto/N`
   - 并发执行
   - 输出按输入顺序稳定
   - 异常不打乱顺序

   这些问题用标准库并发就够了，不需要先把执行器换成 DuckDB。

### 3.3 GPU 暂不进入 runtime 的原因

GPU 看起来“更快”，但 v1 现在不适合，理由更直接：

1. **部署复杂度太高。**
   要处理驱动、CUDA 版本、机器差异、依赖打包，这会直接拉高环境不确定性。

2. **结果一致性风险更高。**
   GPU 路径常常伴随不同数值实现、不同 kernel、不同硬件差异。v1 当前更看重 reproducibility（可复现），不是极限吞吐。

3. **当前工作负载未证明 GPU 值回票价。**
   如果瓶颈主要在任务编排、I/O、Python 函数调度、单 candidate 逻辑，而不是超大矩阵运算，那么上 GPU 不会带来决定性收益。

4. **会让 CI 和本地开发门槛上升。**
   v1 需要的是人人能跑、测试能稳过、错误容易复盘。GPU 会先把门槛抬高。

## 4. ✅ 适合放的位置

### 4.1 Polars 适合的位置

- 数据准备阶段的大表清洗
- 离线 profiling（性能剖析）和基准测试
- 非 runtime 的实验性加速分支
- 将来单独做 `engine v2` 或 `accelerated backend` 的候选实现

### 4.2 DuckDB 适合的位置

- 读取 parquet 做快速检查
- 研究阶段的人肉查询
- run artifacts 的离线分析
- 诊断某一批结果分布、覆盖率、候选通过率

### 4.3 GPU 适合的位置

- 明确存在超大规模矩阵或 rolling/window 计算瓶颈之后
- 单独的实验性加速后端
- 需要批量数值计算、且 CPU profiling 已证明不够用的阶段

## 5. ❌ 当前不适合放的位置

- 不适合把 `Polars` 直接塞进 v1 runtime 主执行链
- 不适合把 `DuckDB` 变成 v1 的核心因子/指标计算执行器
- 不适合让 `GPU` 成为 v1 默认依赖或默认运行要求

更直白一点说：

- v1 现在要的是“唯一口径”
- 不是“多后端并存”

## 6. ⚠️ 风险判断

如果现在就把这些技术拉进 runtime，最容易出现 3 类问题：

1. **行为不一致。**
   同一个输入，在不同后端得到不完全一样的空值、排序、窗口结果。

2. **测试失焦。**
   本来应该验证“顺序稳定和异常稳定”，结果大量时间花在后端差异上。

3. **维护成本提前爆炸。**
   当前仓库还在建立 engine 分层边界。如果现在就上多后端，后续接口会更难收敛。

## 7. 👀 未来再评估条件

满足下面条件后，再考虑把这些能力往 runtime 推进会更合适：

1. 已经有稳定的 v1 baseline。
   也就是：
   - engine 路由稳定
   - 并发行为稳定
   - 顺序稳定
   - 错误行为稳定
   - 测试长期稳定通过

2. 已经完成 profiling（性能剖析）。
   要先明确瓶颈到底在：
   - 数据读取
   - Python 调度
   - 单 candidate 计算
   - 窗口函数
   - 聚合统计

3. 有量化收益门槛。
   比如至少证明：
   - 总耗时下降达到明确比例
   - 内存峰值下降达到明确比例
   - 且结果一致性可验证

4. 有一致性回归测试。
   新后端必须通过：
   - 相同输入输出一致
   - 顺序一致
   - 异常位置一致
   - 边界数据一致

5. 有清晰的后端隔离层。
   也就是先把接口抽稳，再让新后端实现接口，而不是直接把 runtime 主链路改成混合逻辑。

## 8. 🔧 建议落地顺序

建议按下面顺序推进，而不是一步上重技术：

1. 先把 v1 routing 和 ordered parallel 跑稳。
2. 再做 profiling，找到真实瓶颈。
3. 再用离线 benchmark 比较 pandas / Polars / DuckDB / GPU 候选方案。
4. 只有在收益明确且一致性可控时，才进入新的 runtime backend 设计。

## 9. 最终判断

当前阶段最合理的技术取舍是：

- `Polars`：保留为未来候选，不进入 v1 runtime
- `DuckDB`：保留为查询和分析工具，不进入 v1 runtime 主计算链
- `GPU`：保留为后续专项加速方案，不进入 v1 runtime

这不是因为它们“不好”，而是因为 **v1 现在更需要稳定边界、可复现结果、低排障成本**。

先把 runtime 做“稳”，再做“快”，是这阶段更合适的工程顺序。
